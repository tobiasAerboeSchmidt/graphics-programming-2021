Assignment 2 - Weather effects

Description
 In this assignment you will implement part of the dynamic weather simulation effects used in the XBOX360 game Project Gotham Racing 4 (PGR4). The developers have documented how rain and snow effects were created in chapter 5.1 of the book ShaderX 7 (note that I have annotated the paper to make it more relevant for the assignment and correct a code mistake). You can see how the snow and rain effects look like in the game in the trailers below.


The original implementation uses camera facing quads (i.e. pair of triangles) and a texture. As we didn't explain the use of textures in class yet, I recommend that you use points (for slow particles and snow) and lines (for blurred rain) instead (but you can try to use textures if you want a challenge).

Your implementation should use a first person camera with translation and rotation, so that you can explore the environment and visualize the effect that moving the point of view has on the particles. Remember to draw a floor as well as a couple of reference 3D objects, so that it is easy to see when the camera is moving and the direction of movement.

You should refer to both, the original paper as well as this document to implement the program. The following past exercises might prove useful to the development of this assignment:
  • Exercise 2 (drawing particles)
  • Exercise 5 (simple scene, perspective projection and camera control)

You are free to use the code structure you want (i.e. based on the exercises or your own), but you should use the same libraries (opengl, glfw, glad and glm) and make sure that I can run your code by simply adding it to the folder "assignments", running cmake, and building the executable (make sure to make a folder for your source code, shaders, and write - or copy - a cmakelists.txt that can build your project).

Weather effects 1
1. Make it rain
  • Prepare an application that:
    • Draws a simple scene;
    • Allows you to navigate using a first person camera;
    • Can draw particles in addition to the solid scene geometry.
  • Follow the chapter 5.1 up to "listing 5.1.1" to animate the particles, here are a few useful tips:
    • You can use GL_POINTS to draw the individual vertices;
    • You can use the vertex shader built-in variable gl_PointSize to make particles that are close to the camera bigger than particles that are far from the camera, remember to add glEnable(GL_VERTEX_PROGRAM_POINT_SIZE) to your .cpp code to enable the gl_PointSize built-in variable;
    • You don't need to care about acceleration, a constant velocity should reproduce rain fall pretty well since air resistance limits the maximum speed of a rain drop, so this speed is pretty much constant at the time it hits the ground;
    • Remember that the rain drops are in world space, which means that they should be multiplied by the camera view-projection matrix to be visible to our first person camera;
    • You don't need to implement the Perlin noise to control the wind current, it is fine if you use a constant wind velocity instead;
    • Remember that the paper states that the particles have an initial random position that is never modified during the execution of the application (it is only modified in the vertex shader, see annotations in the PDF);
    • You can use the vertex shader built-in variable gl_PointSize to make particles that are close to the camera bigger than particles that are far from the camera, remember to add glEnable(GL_VERTEX_PROGRAM_POINT_SIZE) to your .cpp code to enable the gl_PointSize built-in variable;

Weather effects 3
3. Simulate camera motion blur
  • Modify your program to:
    • Draw lines instead of points, this means that instead of using GL_POINTS you should use GL_LINES;
    • Lines are drawn using 2 consecutive vertices, which means that you should initialize two consecutive vertices with the same random position for each line (remember, the paper states that the particles have an initial random position in the range between [BoxSize, 2*BoxSize] that is never modified during the execution of the application);
    • Follow the instructions of Listing 5.1.2, you can ignore equation 5.1.1 and its implementation in Listing 5.1.2 since you are not drawing camera facing quads;
    • Notice that the code snippets use the uv values, which are either 0 or 1, to find out which vertex in the pair of triangles is being processed. That is, "lerp(topPrev, bottom, uv.y)" will either return "topPrev" or "bottom" instead of a mix of both as the position of the vertex being processed. You also need to make this kind of distinction between the two vertices making up a line in your implementation. A simple way to achieve that is by checking whether the gl_VertexID built-in variable is even or odd (i.e. taking the modulo with 2), so that you set the "gl_Position" to "topPrev" or "bottom" according to that.
    • The current vertex should be placed either at the top or at the bottom (depending on whether that is an even or and odd vertex), which will make up a line that represents the rain drop.

Weather effects 4
 4. Simulate the camera exposure time
  • Notice how weird the rain particles look when the camera rotates. That is because the lines become longer to simulate motion blur, occupying a larger area of the screen. However, on a real camera, motion blur also results in a reduced exposure time of the object to the sensor of the camera (ie. the object is visible to a particular position of the sensor for a much shorter interval of time). To fix that, we must ensure that the color (alpha component) is proportional to the length of the rain particle (which in turn is proportional to camera movement speed), so that the longest the particles is, the more transparent it becomes. To achieve that you need to:
    • Implement the relevant portions of listing 5.1.3;
    • Send the "lenColorScale" to the fragment shader and use it for alpha blending, remember to enable alpha blending with the commands "glEnable(GL_BLEND);" and "glBlendFunc(GL_SRC_ALPHA, GL_DST_ALPHA);"
